# Triage

プロジェクトの issue tracker 上の issue を、triage roles からなる小さな state machine に通す。

この repo が external な pull request を request surface として扱う場合（issue-tracker config を参照）、triage はそれらもカバーする: **PR はコードが添付された issue である** — 同じ roles、同じ states、同じ machine。ただし下記に「for a PR」と marked された、いくつかの差分がある。裸の `#42` は tracker config に従って issue または PR に解決する。

triage 中に issue tracker に投稿される comment や issue は**すべて**、この disclaimer で始まら**なければならない**:

```
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 長持ちする agent brief の書き方

## Roles

2 つの **category** role:

- `bug` — 何かが壊れている
- `enhancement` — 新機能または改善

5 つの **state** role:

- `needs-triage` — maintainer が評価する必要がある
- `needs-info` — reporter からの追加情報待ち
- `ready-for-agent` — 完全に specified され、AFK agent 向けの準備ができている
- `ready-for-human` — 人間による implementation が必要
- `wontfix` — 対応しない

PR の場合、同じ states は添付されたコードに対して読まれる: `ready-for-agent` は brief が添付されており agent が diff に対して次の一歩を進めるべきことを意味し、`ready-for-human` は人間が merge する準備ができていることを意味する。

triage された issue はそれぞれ、ちょうど 1 つの category role と 1 つの state role を持つべきである。state role が矛盾する場合、それを flag し、他の何かをする前に maintainer に尋ねる。

これらは canonical な role 名である — issue tracker で実際に使われる label 文字列は異なる場合がある。マッピングはすでにあなたに提供されているはずである — なければ `/setup-matt-pocock-skills` を実行する。

State transitions: label の無い issue は通常まず `needs-triage` になる。そこから `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` のいずれかに移る。`needs-info` は reporter が返信すると `needs-triage` に戻る。maintainer はいつでも override できる — 通常と異なる transition は flag し、進める前に尋ねる。

## Invocation

maintainer は `/triage` を呼び出し、自然言語で望むことを説明する。要求を解釈して行動する。例:

- "Show me anything that needs my attention"
- "Let's look at #42"（issue または PR）
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 注意が必要なものを示す

issue tracker に問い合わせ、古い順に 3 つの bucket を提示する:

1. **Unlabeled** — 一度も triage されていない。
2. **`needs-triage`** — 評価が進行中。
3. **前回の triage notes 以降に reporter の活動がある `needs-info`** — 再評価が必要。

PR が対象範囲に含まれる場合、これらの bucket に external な PR も含め、各行に `[PR]` または `[issue]` の tag を付ける。discovery は *external* な PR のみを表に出す（何を external とみなすかは tracker config が定義する）— collaborator の作業中の PR は triage の対象ではない。このフィルタは discovery 専用であり、明示的に名指しされた PR は author に関係なく常に triage される。

件数と項目ごとの 1 行要約を示す。maintainer に選ばせる。

## 特定の issue または PR を triage する

1. **context を集める。** issue または PR の全体（本文、comment、label、author、日付。PR なら diff も）を読む。以前の triage notes を parse し、解決済みの質問を再び尋ねないようにする。プロジェクトの domain glossary を使って codebase を探索し、その領域の ADR を尊重する。codebase に対して 2 つのチェックを行う: (a) **redundancy** — 要求された behavior の既存の implementation を、request の言い回しだけでなく domain concept で検索し、どこを見たか報告する。見つかれば、それはすでに implement 済みの `wontfix`（手順 5）である。(b) **prior rejection** — `.out-of-scope/*.md` を読み、この request に似たものを表に出す。

2. **推奨する。** maintainer に category と state の推奨を理由とともに伝え、加えて request に関連する codebase の簡潔な要約 — すでに implement 済みかどうかを含む — を伝える。指示を待つ。

3. **主張を検証する。** grilling の前に、主張が成り立つか確認する。バグなら、reporter の手順から再現する。PR なら、diff が主張通りに動くことを確認する — checkout し、関連するテストやコマンドを実行する。何が起きたか報告する: confirmed（code path 付き）、failed、または detail 不足（強い `needs-info` の signal）。confirmed な検証は、はるかに強い agent brief になる。

4. **必要なら Grill する。** request を肉付けする必要があるなら、`/grilling` と `/domain-modeling` skills を一緒に実行する — 一度に 1 つずつ質問して形にしていき、domain の用語を研ぎ澄まし、決定が固まるにつれてその場で `CONTEXT.md`/ADR を更新する。

5. **結果を適用する:**
   - `ready-for-agent` — agent brief の comment を投稿する（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — agent brief と同じ構造だが、なぜ委譲できないか（判断が必要、外部アクセスが必要、design decision、手動テスト）を記す。
   - `needs-info` — triage notes を投稿する（下記の template）。
   - `wontfix` — close する。comment は*理由*によって異なる:
     - **すでに実装済み** — その変更はすでに codebase に存在する。どこにあるか示す。`.out-of-scope/` には書か**ない**（この KB は *rejected* された request のためであり、作られたもののためではない）。
     - **Rejected（bug）** — 丁寧な説明をしてから close する。
     - **Rejected（enhancement）** — `.out-of-scope/` に書き、comment からそこへリンクし、それから close する（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — role を適用する。部分的な進捗があれば任意で comment する。

## Quick state override

maintainer が「move #42 to ready-for-agent」と言ったら、それを信じて role を直接適用する。これから行うこと（role の変更、comment、close）を確認してから行動する。grilling はスキップする。grilling session なしに `ready-for-agent` へ移す場合、agent brief を書きたいか尋ねる。

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

grilling 中に解決したことはすべて「established so far」の下にキャプチャし、作業が失われないようにする。質問は「もっと情報をください」ではなく、具体的で実行可能なものでなければならない。

## 以前のセッションを再開する

issue や PR に以前の triage notes が存在する場合、それを読み、reporter が未解決の質問に答えていないか確認し、続ける前に更新された状況を提示する。解決済みの質問を再び尋ねない。
