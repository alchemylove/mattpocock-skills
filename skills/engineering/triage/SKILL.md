# トリアージ (Triage)

プロジェクトの issue tracker 上の issue を、小さな triage role の state machine で進める。

triage 中に issue tracker へ投稿するすべての comment または issue は、**必ず** 次の disclaimer で始めること:

```
> *This was generated by AI during triage.*
```

## 参照ドキュメント (Reference docs)

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — durable agent brief の書き方
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` knowledge base の仕組み

## ロール (Roles)

2 つの **category** role:

- `bug` — 何かが壊れている
- `enhancement` — 新機能または改善

5 つの **state** role:

- `needs-triage` — maintainer が評価する必要がある
- `needs-info` — reporter からの追加情報を待っている
- `ready-for-agent` — 完全に仕様化され、AFK agent の準備完了
- `ready-for-human` — 人間による実装が必要
- `wontfix` — 対応しない

triage 済みの issue は、category role を 1 つ、state role を 1 つだけ持つこと。state role が矛盾する場合は flag し、他の何より先に maintainer に確認する。

これらは canonical role 名である — issue tracker で実際に使う label 文字列は異なる場合がある。mapping は提供されているはず — なければ `/setup-matt-pocock-skills` を実行する。

State transitions: label なし issue は通常まず `needs-triage` へ。そこから `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` へ移動する。`needs-info` は reporter が返信したら `needs-triage` に戻る。maintainer はいつでも override できる — 異常に見える transition は flag し、進む前に確認する。

## 呼び出し (Invocation)

maintainer は `/triage` を呼び出し、自然言語で望むことを説明する。リクエストを解釈して行動する。例:

- "Show me anything that needs my attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 注意が必要なものを表示する (Show what needs attention)

issue tracker を query し、3 つの bucket を古い順に提示する:

1. **Unlabeled** — 一度も triage されていない。
2. **`needs-triage`** — 評価進行中。
3. **`needs-info` で、前回の triage notes 以降に reporter activity がある** — 再評価が必要。

件数と issue ごとの 1 行 summary を示す。maintainer に選ばせる。

## 特定 issue の triage (Triage a specific issue)

1. **Gather context.** issue 全体（body、comments、labels、reporter、dates）を読む。prior triage notes を parse し、解決済みの質問を再質問しない。プロジェクトの domain glossary を使ってコードベースを探索し、領域の ADR を尊重する。`.out-of-scope/*.md` を読み、この issue に似た prior rejection を surface する。

2. **Recommend.** category と state の推奨と理由、issue に関連する簡潔な codebase summary を maintainer に伝える。指示を待つ。

3. **Reproduce (bugs only).** grilling の前に再現を試みる: reporter の手順を読み、関連コードを trace、test や command を実行。結果を報告 — 成功した repro と code path、失敗した repro、または不十分な詳細（強い `needs-info` シグナル）。確認された repro はより強い agent brief になる。

4. **Grill (if needed).** issue の肉付けが必要なら、`/grill-with-docs` セッションを実行する。

5. **Apply the outcome:**
   - `ready-for-agent` — agent brief comment を投稿（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — agent brief と同じ構造だが、委任できない理由を記載（judgment calls、external access、design decisions、manual testing）。
   - `needs-info` — triage notes を投稿（下記 template）。
   - `wontfix` (bug) — 丁寧な説明の後、close。
   - `wontfix` (enhancement) — `.out-of-scope/` に書き、comment からリンクし、close（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — role を適用。部分的進捗があれば optional comment。

## クイック state override (Quick state override)

maintainer が "move #42 to ready-for-agent" と言ったら、信頼して role を直接適用する。これから行うこと（role changes、comment、close）を確認してから実行。grilling は skip。grilling なしで `ready-for-agent` に移す場合、agent brief を書くか確認する。

## needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

grilling で解決した内容はすべて "established so far" に capture し、作業を失わない。質問は具体的で actionable であること。「please provide more info」ではない。

## 前回セッションの再開 (Resuming a previous session)

issue に prior triage notes があれば読み、reporter が未解決の質問に答えたか確認し、続ける前に更新された状況を提示する。解決済みの質問を再質問しない。
