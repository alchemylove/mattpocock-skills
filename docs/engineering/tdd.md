## 何をするか (What it does)

`tdd` は、機能を構築したりバグを修正したりする際、テストファーストで、一度に1つの振る舞いずつ、red-green ループを通してコードを引き出していきます。

すべてのテストを最初にまとめて書くことは**しません**。テストを先にまとめて書く（「水平方向のスライス」）と、*想像上の*振る舞いに対するテストが生まれます — それらは物事の形をチェックするだけで、実際の変更に対して感度を失います。`tdd` の代わりに vertical slice を取ります — 1つのテスト、それを通すためのちょうど十分なコード、次のテスト、という具合に、各サイクルは直前のサイクルから学んだことに基づきます。テストは公開インターフェースだけを対象とするので、下にある実装はテストを動かすことなく変化できます。

## いつ使うか (When to reach for it)

`/tdd` と入力するか、タスクに合致すればエージェントが自動的に使います — テストファーストで機能を構築したりバグを修正したりするとき、あるいは「red-green-refactor」と言ったときです。

構築すべき具体的な振る舞いがあり、リファクタリングに耐えるテストが欲しいときに使ってください。振る舞いがまだ固まっていないなら、まず仕様を固めてください — そのためには [to-prd](https://aihero.dev/skills-to-prd) を使います。作業が本当にテストではなくインターフェースの形についてのものであれば、[codebase-design](https://aihero.dev/skills-codebase-design) を使ってください。`tdd` は計画時に deep-module の語彙を得るためにこれを呼び出します。

## Red-Green、一度に1スライスずつ

主軸となるアイデアは **red-green ループ** です — 1つの失敗するテストを書き（red）、それを通すためのちょうど十分なコードを加え（green）、次の振る舞いについて繰り返します — 各サイクルは直前のサイクルから学んだことに基づきます。最初のサイクルは **tracer bullet** です — 外側へ構築していく前に、1つのパスがエンドツーエンドで動作することを証明する1つのテストです。たった今そのコードを書いたばかりなので、どの振る舞いが重要で、どう検証すればよいかを正確に把握しています — まだ理解していないテスト構造にコミットして自分のヘッドライトを追い越すことはありません。

テストを誠実に保つ2つのルールがあります。良いテストは仕様のように読め（「ユーザーは有効なカートでチェックアウトできる」）、公開 API を通じて実際のコードパスを行使するので、内部関数のリネームがそれを壊すことはありません。そして期待値は独立した信頼できる情報源 — 既知の正しいリテラル、実例、仕様 — に由来し、コードが計算するのと同じ方法で再計算されることは決してありません。それをやってしまうと、構造的に必ず通ってしまい、何も教えてくれない **tautological（同語反復的な）** テストになります。

リファクタリングはスイートが green になってから初めて行います。red の間は決して行いません。

## うまく機能しているかの目安 (It's working if)

- 1つのテストを書き、それを通し、それからようやく次のテストを書く — テストのバッチの後にコードのバッチが続くのではない。
- テストは内部実装ではなく振る舞いを名付けており、内部のリネームに耐えられる。
- 期待値は仕様由来のリテラルであり、コードが導出するのと同じ方法で導出された数値ではない。

## 全体の中での位置づけ (Where it fits)

`tdd` は main build chain がコードを書くために実行する red-green ループです。

```txt
grill-with-docs → to-prd → to-issues → implement → code-review
```

[implement](https://aihero.dev/skills-implement) は chain のビルドステップであり、各チケットをテストファーストで構築するために内部で `tdd` を駆動してから [code-review](https://aihero.dev/skills-code-review) へ引き渡します。つまり `tdd` はそれ自体の1ステップというより、そのステップの中のエンジンです。完全な仕様がなくても具体的な振る舞いを構築したいときは、直接これを使うこともできます。もう1つの隣人は [codebase-design](https://aihero.dev/skills-codebase-design) で、テストする価値のある deep-module の seam を見つけるためにこれに頼ります。どの skill や flow が合うか迷ったときは、[ask-matt](https://aihero.dev/skills-ask-matt) が導いてくれます。
